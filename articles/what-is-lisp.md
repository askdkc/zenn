## TL;DR

LISPの良さは「問題を式として置ける」ことにある。

人間は問題を言葉で理解し、コンピュータは最終的に0と1で処理する。
その間には、構造が必要になる。

多くの言語ではコードは処理系の中でASTへ変換されるが、その構造は普段隠れている。
LISPではプログラムがリストとして現れ、コードとデータの距離が近い。

だからLISPでは、問題に合わせて小さい関数や言葉を育てられる。

DBからJSONを返すWeb APIでも、DBの形をそのまま漏らすのではなく、APIが外に見せる形を作る必要がある。
その「形を作る」感覚を、LISPはかなり素直に見せてくれる。

## これは何？

LISPという、1958年頃からあるプログラミング言語についてのお話です。

1958年。

古い。
かなり古い。

「古代兵器」と言うと怒られそうですが、実際かなり古い。
でも、古いから弱いかというと全然そうではない。

むしろ問題は逆である。

古いのに、いまだに妙に強い。
そして強さの方向性がだいぶ変である。

なぜか。

LISPは単なるプログラミング言語というより、**世界と人間とコンピュータを繋ぐ数式**だからである。

いきなり怪しい。

「世界と人間とコンピュータを繋ぐ数式」とか言うと、急に情報商材っぽい。
壺とか売りそう。

でも、これはかなり現実的な話である。

人間は世界をそのまま理解していない。
自分の言葉で理解している。

コンピュータも世界をそのまま理解していない。
最終的には0と1で処理している。

その間に橋が必要になる。

その橋が、式であり、構造であり、プログラムである。
LISPは、その橋の形がやたら見えやすい。

ここで言う「数式」とは、算数だけの話ではない。
問題の一部を切り出し、名前を付け、関係を作り、コンピュータが評価できる形に置くことだ。

## コンピュータは人間の言葉を話していない

まず前提として、コンピュータは人間の言葉を話していない。

コンピュータが最終的に扱っているのは0と1である。
バイナリである。

人間はこう言う。

> 私は、プログラミングが、好きだ。

コンピュータはこれをそのまま理解しない。

「私」も知らない。
「プログラミング」も知らない。
「好きだ」も知らない。
助詞の「は」と「が」の違いも知らない。
空気も読まない。
文脈も察しない。
「まあ分かるでしょ」も通じない。

当然である。

コンピュータは人格者ではない。
電気で動く石である。

なので、人間の言葉をコンピュータに伝えるには、どこかで形を変える必要がある。

たとえば、

> 私は、プログラミングが、好きだ。

という文を、かなり単純化して、こういう構造として置いてみる。

```lisp
(:sentence
  (:topic "私")
  (:target "プログラミング")
  (:predicate "好きだ"))
```

これは正確な日本語解析ではない。
日本語の構文解析や意味解析は、こんなに単純ではない。

ここでは「文も構造として置き直せる」という感覚だけ見ればよい。

人間の言葉をそのまま渡すのではなく、構造として表現し直す。
これが重要である。

```text
人間の言葉
  ↓
構造
  ↓
データ
  ↓
バイナリ
```

この階段がある。

プログラミングとは、単にコンピュータに命令を書くことではない。
人間が見ている問題を、コンピュータが処理できる構造に変える作業である。

## プログラムもそのまま実行されているわけではない

ここで、もう一つ重要な話がある。

人間が書いたプログラムも、そのままCPUに読まれているわけではない。

たとえば人間は、こう書く。

```lisp
(+ (* 10 5) (/ 2 13))
```

人間には、これは計算式に見える。

だが、コンピュータの内部では、これがそのまま魔法のように実行されているわけではない。

人間が書いたプログラムは、まず処理系によって構文の形に分解される。
多くの言語では、その構文はAST、つまり**抽象構文木**として扱われる。

そのASTはさらに変換され、最適化され、最終的にはCPUが実行できる機械語、つまり0と1のビット列へ落とし込まれる。

ざっくり言えば、こうである。

```text
人間が書いたコード
  ↓
パースされる
  ↓
AST（抽象構文木）になる
  ↓
内部表現へ変換される
  ↓
機械語へ落とされる
  ↓
CPUが実行する
```

多くの言語では、このASTはコンパイラやインタプリタの内側に隠れている。

プログラマはコードを書く。
だが、そのコードが処理系の中でどのような木として読まれているかを、普段は直接触らない。

もちろん、ASTを取り出すライブラリはある。
コンパイラ内部を覗く方法もある。
言語によってはメタプログラミングもある。

ある。
それはそう。

でも、普通のプログラミングでは、ASTは「処理系の中の話」である。

LISPは違う。

LISPでは、プログラムそのものがリストとして現れる。

```lisp
(+ (* 10 5) (/ 2 13))
```

これは式である。
同時に、リストでもある。

引用すれば、計算される前の形をデータとして持てる。

```lisp
'(+ (* 10 5) (/ 2 13))
```

これは、もはや「実行される命令」ではない。

`+` と `(* 10 5)` と `(/ 2 13)` を要素に持つリストである。

つまり、プログラムの形そのものが、プログラマの手元にデータとして出てくる。

ここがLISPのおかしいところであり、強いところである。

厳密に言えば、LISPのS式が処理系内部のASTそのものだ、という雑な言い方は危ない。

処理系の内部では、さらに別の表現に変換される。
最適化もされる。
実装ごとの都合もある。

だから、

> LISPのコード = 処理系内部のASTそのもの

と言い切ると、少し雑である。

でも、

> LISPのコードは、一般的な言語でコンパイラ内部に隠れがちなASTに近い構造を、プログラマが普通にデータとして扱える形で表に出している

なら、かなり本質を突いている。

だからLISPでは、ただプログラムを書くのではない。

プログラムを書く前に、問題を表すための言語そのものを作ることができる。

## ASTの話は、コンパイラだけの話ではない

ASTの話は、コンパイラだけの話ではない。

重要なのは、複雑なものをそのまま扱わず、**意味のある構造に置き直す**という発想である。

コンパイラは、人間が書いたコードをASTにする。
Web APIを書くプログラマは、DBの行をAPIレスポンスの構造にする。

やっていることは違う。
でも、どちらも「入力を、そのまま扱うのではなく、意味のある形に置き直す」作業である。

LISPは、この「形に置き直す」作業と相性がいい。

なぜなら、LISPではコードもデータも、リストという見える構造として扱いやすいからだ。

## 人間が全部コンピュータに合わせる必要はない

普通、プログラミングを学ぶと、人間がコンピュータに合わせる。

この文法で書け。
この型で渡せ。
この構文で分岐しろ。
この形で関数を呼べ。
このフレームワークの流儀に従え。

もちろん、それは必要である。

コンピュータに伝える以上、最低限の約束は守らないといけない。
0と1しか理解しない機械に、人間の気分をそのまま渡しても仕方ない。

だが、LISPはそこで少し違うことをする。

人間がコンピュータにすべて合わせるのではない。
一度、コンピュータが理解できる構造に歩み寄る。
そのうえで、人間の思考に合う言葉を作り直す。

これがLISPの魔法である。

いきなり大げさに聞こえるかもしれない。

なので、今どきのWeb APIで考える。

DBからデータを取り出す。
APIでJSONを返す。
フロントエンドがそれを表示する。

はい、いつものやつ。

```text
DB
  ↓
API
  ↓
JSON
  ↓
Frontend
```

LaravelでもRailsでもDjangoでもGoでもNodeでも、だいたいこれをやっている。

一見すると、ただのWeb開発である。

だが、ここにはプログラミングの本質がかなり詰まっている。

## DBの形は、人間が見たい形ではない

たとえば、ブログ記事APIを考える。

DBにはこう入っている。

```text
articles
- id
- title
- body
- user_id
- status
- published_at
- created_at
- updated_at

users
- id
- name
- email
- password_hash

tags
- id
- name

article_tag
- article_id
- tag_id
```

DBとしては正しい。
正規化されている。
リレーションも分かる。
検索もしやすい。
更新もしやすい。

でも、フロントエンドが欲しいのはこれではない。

フロントエンドが欲しいのは、たぶんこういうJSONである。

```json
{
  "id": 1,
  "title": "LISP：世界と人間とコンピュータを繋ぐ数式",
  "author": {
    "id": 10,
    "name": "circleback"
  },
  "tags": ["lisp", "api", "json"],
  "published_at": "2026-05-17"
}
```

つまり、DBの形と、APIが返す形は違う。

DBは保存しやすい形。
APIレスポンスは利用しやすい形。
JSONはフロントエンドに渡しやすい形。
UIは人間に読ませやすい形。

それぞれ目的が違う。

ここを混ぜると、あとで壊れる。

初心者はよくこう考える。

「DBから取って、そのままJSONで返せばいいじゃん」

まあ、動く。
小さいうちは動く。

Laravelなら、こういうことができる。

```php
return Article::with('user', 'tags')->find($id);
```

はい便利。

なんかそれっぽいJSONが返る。
便利過ぎだろ。

もちろんLaravelにはAPI Resourceがある。
だから「Eloquentモデルをそのまま返すな」で話を終わらせるのは浅い。

本当に見るべきなのは、Resourceを使っているかどうかではない。
そのResourceが、外に見せたい概念をちゃんと表しているかである。

そこを見ないまま、DBの形を少し整えただけのJSONを返し続けると問題が出る。

DBのカラム名が外部API仕様になる。
内部都合のリレーションが漏れる。
フロントエンドがDB構造に依存する。
あとでDB設計を変えにくくなる。
見せたくないフィールドが混ざる。
ResourceがEloquentモデルの化粧箱になる。
一覧用、詳細用、管理画面用、公開API用の概念が混ざる。
APIレスポンスの責務がどこにあるのか分からなくなる。

DBの形は、保存の都合である。
APIの形は、利用者の都合である。

だから一度、こう考える必要がある。

```text
db rows
  ↓
application data
  ↓
api response
  ↓
json
```

いきなりDBからJSONへ直結しない。
一回、「このAPIが返すべき形」を作る。

ここでLISP的な見方が効く。

## APIは、システムが外に見せる概念である

APIは、単にDBの中身を外に出す穴ではない。

そんな雑な穴を開けると、あとで困る。

APIは、外の世界に対して、

> このシステムでは、この概念をこう見せます

と宣言する場所である。

記事とは何か。
ユーザーとは何か。
注文とは何か。
検索結果とは何か。
通知とは何か。
コメントとは何か。

これをJSONの形で表現している。

つまりAPIレスポンスは、単なるデータではない。
外部仕様である。

たとえば、記事詳細APIがこう返すとする。

```json
{
  "id": 1,
  "title": "LISP：世界と人間とコンピュータを繋ぐ数式",
  "author": {
    "id": 10,
    "name": "circleback"
  },
  "tags": ["lisp", "programming", "api"],
  "summary": "LISPは世界を式にするための言語である",
  "published_at": "2026-05-17"
}
```

これはただのJSONではない。

このシステムでは、記事にはIDがある。
タイトルがある。
著者がいる。
タグがある。
要約がある。
公開日がある。

そういう概念として記事を見せている。

DBには `user_id` しかないかもしれない。
タグは中間テーブルかもしれない。
要約は本文から生成しているかもしれない。
公開日はDB上ではtimestampかもしれない。

でも、API利用者にはそんな内部事情をそのまま渡さない。

外に出す形を決める。

これがAPI設計である。

## LISPは「返したいJSONの形」を先に見せてくれる

ここで出すS式は、厳密なCommon Lispのデータ設計ではなく、**構造を見せるための疑似的なS式**である。

LISP的に考えると、まず返したい形をリストとして描いてしまう。

```lisp
(:article
 (:id 1
  :title "LISP：世界と人間とコンピュータを繋ぐ数式"
  :author (:id 10 :name "circleback")
  :tags ("lisp" "programming" "api")
  :summary "LISPは世界を式にするための言語である"
  :published-at "2026-05-17"))
```

この時点で、API設計のかなりの部分が見えている。

必要な項目は何か。
ネストするのか。
配列なのか。
文字列なのか。
IDを返すのか。
表示名を返すのか。
内部のカラム名をそのまま出すのか。
外向きの名前に変えるのか。

つまりLISPは、JSONを吐く前に、**返したい形**を見せてくれる。

ここで大事なのは、LISPでWeb APIを書け、という話ではない。

もちろん書いてもいい。
でも本題はそこではない。

LISPを学ぶと、他の言語でAPIを書いている時にも、

> このAPIはどんな形を返すべきなのか？

を先に考える癖がつく。

これはかなり効く。

## 人間の問題を式にする

「記事詳細画面が欲しい」

人間はこう言う。

でも、それはまだコンピュータには曖昧すぎる。

記事とは何か。
タイトルは必要か。
本文は必要か。
著者はIDだけでいいのか、名前も要るのか。
タグはIDが要るのか、名前だけでいいのか。
公開日時はどの形式にするのか。
下書き記事は返していいのか。
ログインユーザーによって見える項目は変わるのか。
管理者には追加情報を返すのか。
一般ユーザーには何を隠すのか。

これを言語化しないと、APIは作れない。

つまり、人間側がまず問題を理解する必要がある。

ただし、これは答えを全部知るという意味ではない。

数学で言えば、答えを暗算できる必要はない。

```text
10 × 5 + 2 / 13
```

この式を作れればいい。
答えはコンピュータが計算する。

APIでも同じである。

```text
記事IDを受け取る
  ↓
記事を取得する
  ↓
公開状態を確認する
  ↓
著者を付ける
  ↓
タグを付ける
  ↓
外に出してよい形に整える
  ↓
JSONで返す
```

この式を作れればいい。

LISPなら、こういう感覚になる。

```lisp
(article->api-response article)
```

もう少し分けるならこう。

```lisp
(make-api-response-for-article
  :id (article-id article)
  :title (article-title article)
  :author (author->api-response (article-author article))
  :tags (mapcar #'tag-name (article-tags article))
  :summary (article-summary article)
  :published-at (format-date (article-published-at article)))
```

ここに出てくる `article->api-response`、`make-api-response-for-article`、`author->api-response`、`format-date` などは、Common Lisp標準の関数ではない。
自分の問題に合わせて、こういう関数を作るという話である。

また、`article->api-response` のような名前に出てくる `->` は特別な演算子ではない。
関数名の一部である。
「article から api-response へ変換する」くらいの意味を持たせた名前だ。

`mapcar` は、リストの各要素に関数を適用して、新しいリストを作る関数である。
ここでは、`article-tags` で得たタグ一覧それぞれに `tag-name` を適用し、タグ名だけのリストを作っている。

細部は今は分からなくていい。

大事なのは、何を返すのか、何を隠すのか、何を変換するのか、どこで整形するのかが見えていることだ。

問題が式になっている。

## 名前を付けると、考えられるようになる

最初は、ただDBから値を取るだけである。

```lisp
(article-title article)
```

まあ分かる。

でも、実際のAPIはそれだけでは済まない。

日付を整形する。

```lisp
(format-date published-at)
```

著者をAPIレスポンス用に変える。

```lisp
(author->api-response author)
```

記事全体をAPIレスポンスにする。

```lisp
(article->api-response article)
```

APIレスポンスをJSONにする。

```lisp
(api-response->json response)
```

最初は「なんか面倒な処理」だったものに名前が付く。

`format-date`  
`author->api-response`  
`article->api-response`  
`api-response->json`

名前が付くと、考えられるようになる。
考えられるようになると、組み替えられる。
組み替えられるようになると、自分の道具になる。

人間は、名前のないものをうまく扱えない。

「なんか変なやつ」では考えられない。
「APIレスポンス整形」なら考えられる。
「公開日フォーマット」なら考えられる。
「著者のAPIレスポンス表現」なら考えられる。
「DBモデルからAPIレスポンスへの変換」なら考えられる。

名前を付けるたびに、曖昧だったものが少しずつ固まっていく。

固まったものは、持てる。
持てるものは、動かせる。
動かせるものは、組み替えられる。

これが気持ちいい。

## DB、API、JSONはそれぞれ違う言葉を話している

DBはDBの言葉を話す。

テーブル。
カラム。
主キー。
外部キー。
インデックス。
正規化。
JOIN。

APIはAPIの言葉を話す。

エンドポイント。
リソース。
レスポンス。
ステータスコード。
認可。
ページネーション。
フィルタ。

フロントエンドはフロントエンドの言葉を話す。

表示。
状態。
コンポーネント。
フォーム。
一覧。
詳細。
ローディング。
エラー。

人間は人間の言葉を話す。

記事。
著者。
タグ。
公開日。
下書き。
検索結果。
通知。

そしてコンピュータは、最終的に0と1を処理している。

全部違う。

だから翻訳が必要になる。

```text
人間の問題
  ↓
アプリケーションの概念
  ↓
DBから取得する形
  ↓
APIとして返す形
  ↓
JSON
  ↓
フロントエンドの表示
```

この翻訳を雑にやると、あとで壊れる。

DB都合のJSONが出る。
フロントエンドがDB構造に依存する。
APIの互換性が壊れる。
Controllerが太る。
モデルが外部仕様を背負う。
レスポンスの形を変えたら画面が壊れる。

だから、どこかで明示的に「形」を作る必要がある。

LISPはこの「形を作る」感覚が強い。

DB行を記事にする。
記事をAPIレスポンスにする。
APIAPIレスポンスをJSONにする。

```lisp
(db-row->article row)
(article->api-response article)
(api-response->json response)
```

名前が付いている。
責務が分かれている。
後で変えられる。

はい便利。

## 反論：それ他の言語でもできるよね？

できる。

Pythonでもできる。
Rubyでもできる。
PHPでもできる。
Goでもできる。
JavaScriptでもできる。

Laravelなら API Resource がある。
Railsなら serializer がある。
Django REST Framework なら serializer がある。
GoならstructにJSONタグを付ける。
TypeScriptなら型を作る。

できる。
普通にできる。

この反論は正しい。

ただし、正しいけど、論点が少しズレている。

問題は「できるか」ではない。
「自然にそう考えられるか」である。

どの言語でも構造は作れる。
どの言語でも関数は作れる。
どの言語でも中間データは作れる。
どの言語でもASTを扱う方法はある。

だが、LISPはコードとデータの距離が近い。

式もリスト。
データもリスト。
構造もリスト。
ASTに近い形もリストとして見える。

だから、問題を構造として見ることと、コードを書くことの距離が近い。

これは地味に効く。
後から効く。

LISPを触った後に他の言語を書くと、

「このEloquentモデル、そのままJSONで返すのは雑では？」  
「ここで一回、レスポンス用の形にした方がよくないか？」  
「この処理、名前を付けた方が意図が見えるのでは？」  
「DBの形とAPIの形が癒着していて、仕様変更で壊れるのでは？」  
「このJSONは本当に利用者が見たい概念を表現できているのか？」

みたいな嫌な目が育つ。

見えなくてよかったものが見える。
つらい。

でも、見えるから直せる。

## マクロ？それは後でいい

LISPの話をすると、すぐマクロが出てくる。

マクロは確かに強い。
LISPの代表的な武器でもある。

だが、初心者にいきなりマクロを見せて、

> ほら、LISPは言語を拡張できるんですよ！

とかやるのは、だいたい悪手である。

初心者には魔法というよりインチキに見える。

なので順番が違う。

まず問題を見る。
次に言葉にする。
それを式にする。
小さい関数を書く。
REPLで試す。
壊す。
直す。
また試す。
同じ処理が増えたら名前を付ける。
名前が増えたら、少し整理する。
それでも足りなくなったら、そこで初めてマクロを考える。

マクロは最初に出すものではない。
最後に出てくるご褒美である。

最初からご褒美を出すと、話が壊れる。

LISPの良さは、いきなり言語を作れることではない。
小さい関数を積み上げた結果、自分の問題に合う言葉が自然に育ってくるところにある。

## LISPで学ぶべきは、括弧ではなく世界の式化

LISPを仕事で使うかどうかは、正直どうでもいい。

いや、使えたら楽しい。
REPLで試しながら関数を育てるのはかなり楽しい。
Emacsを触っている人なら、Elispでその一部は味わえる。

でも、仮に仕事ではPHPを書いているとしても、LISPを学ぶ意味はある。

Laravelを書いていても、こう考えるようになる。

「このAPI Resource、本当に画面が欲しい概念を表しているのか？」  
「Resourceを使っているだけで、API設計をした気になっていないか？」  
「一覧用、詳細用、管理画面用、公開API用のレスポンスを同じ形で済ませようとしていないか？」  
「このCollection操作、名前のある関数にした方が意図が見えるのでは？」  
「このJSONは、Eloquentモデルの都合ではなく、利用者の都合になっているのか？」

PostgreSQLを書いていても、こう考える。

「このVIEWは何の概念に名前を付けているのか？」  
「この検索インデックスは、人間のどの探し方を表現しているのか？」  
「この中間テーブルは、汚い入力を検品する場所として機能しているのか？」  
「この集計結果はAPIに返す前に、もう一段意味のある形にした方が良くないか？」

フロントエンドを書いていても、こう考える。

「このAPIレスポンスは、そのまま画面状態にしていいのか？」  
「表示用の形に変換した方がコンポーネントが単純になるのでは？」  
「この状態管理、サーバーの都合とUIの都合が混ざっていないか？」

Emacsをいじっていても、こう考える。

「この設定、毎回コピペしてるなら関数にした方がいいのでは？」  
「このhookの設定、モードごとの意図が見えるように分けた方がいいのでは？」  
「この不具合、まず関数単位で評価して犯人を探せば良くない？」

はい便利。

LISPは、LISPを書いている時だけ効くものではない。
他の言語に戻った時に、変な後遺症として効く。

問題の形が見える。
名前のない処理が気になる。
入力と出力の直結が怖くなる。
DBとAPIの癒着が気になる。
中間の構造を置きたくなる。
小さい関数を育てたくなる。

これは良い後遺症である。

## 便利過ぎだろ？

LISPを学ぶと、プログラミングの見え方が少し変わる。

コードを書くことが、ただの命令作業ではなくなる。

DBからJSONを返すことが、ただのデータ出力ではなくなる。
APIを作ることが、「このシステムでは、この概念をどう外に見せるのか」を決める作業に見えてくる。

記事とは何か。
ユーザーとは何か。
検索結果とは何か。
注文とは何か。
通知とは何か。
公開とは何か。
下書きとは何か。
外に出してよい情報とは何か。

これらに名前を付ける。
形を決める。
式にする。
コンピュータに評価させる。

そうすると、コンピュータは答えを返す。

人間は答えを全部知っている必要はない。
問題を式にできればいい。

これがLISPである。

人間がコンピュータにすべて合わせるのではない。
コンピュータが理解できる形に歩み寄ったうえで、人間の思考に合う言葉を作り直す。

その言葉で問題を表す。
その式をコンピュータが評価する。
結果を人間が受け取る。

LISPは、その橋の形がそのまま見える。

これが楽しくないなら、たぶんプログラミングのかなり美味しいところをまだ食べていない。
