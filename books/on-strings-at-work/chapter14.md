---
title: 終章――汚い文字列地獄をDB連携可能な業務フローに変える
---
数週間後、ミナの仕事は「Excel転記だけを手伝うインターン」ではなくなっていた。

彼女は、文字列地獄を業務フローへ変えた。

```text
メール
  ↓ Emacsで観察
必要行抽出
  ↓ Common Lispで正規化
plist / row
  ↓ CSV / JSON
staging table
  ↓ SQLで検査
本テーブル
  ↓ API / レポート / 画面
```

高村は言った。

「最初は、ただのテキスト整形だと思ってた。」

ミナは笑った。

「私もです。でも、文字列は入口でした。」

黒瀬が言った。

「入口？」

「はい。メールも、CSVも、ログも、JSONも、最初は文字列です。でも、そのままでは業務に使えません。観察して、正規化して、構造にして、検査して、人間に返して、DBへ流す。そこまでやって、初めて業務フローになります。」

最後にミナは、チームのWikiにこう書いた。

```text
文字列をそのまま信じない。
まず見る。
揺れを見つける。
小さく整える。
名前と値に分ける。
リストにする。
人間に戻す。
それが、文字列地獄を業務フローに変える順番です。
```

さらに、その下に実務用の手順を続けた。

```text
汚い文字列地獄をDB連携可能な業務フローに変える手順

1. Emacsで観察する
2. 汚れを分類する
3. 小さな文字列操作を書く
4. 入力例・出力例・失敗例を残す
5. リファクタリングして関数名をつける
6. JSONの入れ子はpathで辿る
7. rawとnormalizedを残す
8. PostgreSQL staging tableに入れる
9. SQLで検査する
10. 本当に安全な行だけ本テーブルへ流す
11. 人間判断が必要なものは止めて返す
```

それは、派手な魔法ではなかった。

けれど、会社の仕事を確かに前へ進める技術だった。
